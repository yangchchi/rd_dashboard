# AI 研发流水线开发执行计划

## 1. 计划目标

在当前项目基础上，分阶段落地“可执行流水线引擎”，实现：
- 自动执行编码计划、构建、测试、质量检查、部署。
- 平台端步骤级实时可视化与人工干预。
- 质量门禁与审计闭环。

## 2. 范围定义

### 2.1 本期范围（建议 6-8 周）
- 后端：编排引擎、步骤执行、日志/工件/门禁数据模型。
- 前端：步骤级看板、实时日志、门禁面板、失败诊断与重试。
- 集成：Git 提交、CI 触发、测试报告回传。

### 2.2 非本期范围
- 多租户资源隔离。
- 复杂审批工作流引擎。
- 全量 FinOps 成本治理平台。

## 3. 里程碑与交付物

## M1（第 1-2 周）：基础编排与数据底座

目标：
- 打通 PipelineRun / StepRun 的创建、状态流转、持久化。

交付物：
- 新增表结构：
  - `rd_pipeline_runs`
  - `rd_pipeline_step_runs`
  - `rd_pipeline_step_logs`
  - `rd_pipeline_artifacts`
  - `rd_pipeline_quality_gates`
- 后端 API：
  - 创建运行、查询运行、步骤查询、日志查询。
- 状态机 v1：
  - `queued -> running -> succeeded/failed/cancelled`
  - step 级 `pending/running/succeeded/failed/skipped`

验收：
- 能创建一条运行记录并推进多个步骤状态。
- 前端可查看步骤列表和基础日志。

## M2（第 3-4 周）：执行器接入与真实步骤

目标：
- 将“模拟流水线”升级为“真实执行”。

交付物：
- Runner Gateway：
  - Codegen Runner（先接现有能力接口，后替换 OpenClaw）
  - Build/Test Runner（执行脚本并回写结果）
- 集成能力：
  - Git 分支/commit 回写
  - 测试报告标准化回传
- 失败重试机制：
  - 可配置重试次数与退避策略

验收：
- 流水线可自动执行 `plan -> codegen -> build -> test`。
- 测试报告在平台可见，失败可重试。

## M3（第 5-6 周）：质量门禁与部署闭环

目标：
- 增加质量门禁与部署/回滚能力，形成交付闭环。

交付物：
- Quality Gate：
  - coverage、lint、security 等规则
  - 阻断与 override（含审计）
- Deploy Runner：
  - dev/staging 部署
  - 失败自动回滚
- 审计日志：
  - 记录关键动作（重试、取消、回滚、override）

验收：
- 质量不达标时阻断后续步骤。
- deploy 失败可自动回滚并在 UI 清晰展示。

## M4（第 7-8 周）：体验增强与治理

目标：
- 提升可观测性与团队协作效率。

交付物：
- 前端增强：
  - 步骤时间线、关键错误诊断、日志过滤检索
  - 质量趋势与历史运行对比
- 通知机制：
  - 关键事件推送（失败、阻断、完成）
- 运行指标：
  - 成功率、平均耗时、失败分布

验收：
- 产品/技术经理能通过页面独立定位失败原因并决策下一步动作。

## 4. 后端任务拆解（建议）

### 4.1 模块拆分
- `pipeline-orchestrator.module`
- `pipeline-runner.module`
- `pipeline-quality-gate.module`
- `pipeline-events.module`

### 4.2 核心任务
- 定义状态机与转移校验。
- 实现运行与步骤数据写入规范（幂等 key、attempt）。
- 接入消息队列（BullMQ + Redis）处理异步步骤。
- 统一错误码体系（业务错误 / 基础设施错误 / 可重试错误）。

### 4.3 API 契约任务
- 新增运行级 API 与步骤级 API。
- 为前端提供分页日志与工件查询能力。
- 标准化测试报告和门禁结果结构。

## 5. 前端任务拆解（建议）

### 5.1 页面与交互
- 流水线看板改为 Run + Step 双层展示。
- 详情页加入步骤时间线与门禁结果。
- 日志页支持按步骤过滤、关键字搜索、自动滚动与暂停。

### 5.2 状态同步
- 实时通道：SSE/WebSocket。
- 降级轮询：通道异常时自动切换轮询。

### 5.3 操作能力
- 步骤级重试、运行级取消、失败后回滚。
- 高风险操作二次确认 + 权限校验。

## 6. 团队分工建议

- 架构/后端负责人：状态机、编排引擎、数据模型。
- 平台后端工程师：Runner 接入、CI/CD Adapter、质量门禁。
- 前端工程师：流程可视化、实时日志、失败诊断 UI。
- QA：自动化测试基线、回归策略、门禁阈值验证。
- DevOps：Runner 环境、部署策略、监控报警。

## 7. 风险清单与应对

- 代码代理输出不稳定：
  - 增加“计划审阅 + 变更审查 + 门禁”三道保护。
- 外部工具波动（Git/CI）：
  - 适配器隔离 + 重试 + 熔断降级。
- 日志与工件膨胀：
  - 分层存储与归档策略（热/冷数据）。
- 执行环境安全风险：
  - 沙箱隔离、最小权限凭证、命令白名单。

## 8. 验证与测试计划

- 单元测试：
  - 状态机转移、门禁判定、错误码映射。
- 集成测试：
  - run 创建到完成全链路；失败重试与回滚链路。
- 压力测试：
  - 并发运行数、日志吞吐、队列积压恢复。
- 回归测试：
  - 需求/PRD/Spec/流水线/验收主链路无回退。

## 9. 上线策略

- Phase A：灰度开启（仅内部角色可用）。
- Phase B：部分需求线启用真实执行，其余保留旧模式。
- Phase C：全量切换并关闭旧的模拟流程。

上线门槛：
- 核心链路成功率 >= 95%。
- 失败任务可诊断率 >= 99%（必须有错误码与关键日志）。
- 质量门禁正确拦截率 >= 99%。

## 10. 完成定义（Definition of Done）

- 架构文档、API 契约、数据模型齐备并评审通过。
- 流水线全链路可在平台端可视化追踪。
- 关键步骤支持重试/取消/回滚且有审计。
- 自动化测试与门禁规则在生产可执行。
- 监控告警覆盖核心异常场景。
